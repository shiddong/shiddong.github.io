资源表示的metadata

资源的表示(representation)旨在通过协议容易传输的格式、并伴随着元数据(metadata)以反映出资源「现在」「过去」「预期」的状态(state)。

资源表示的元数据(即Representation header fields)通常包括：

  1. 内容类型(Content-Type)
  2. 内容编码(Content-Encoding)
  3. 内容语言(Content-Language)
  4. 内容位置(Content-Location)
    …

Content-Type

Content-Type 是用来表示资源的「媒体类型」(Media-Type)，即定义资源的数据格式(data format)，以供接收者使用相应的方式处理。

在传递数据给后端时，即在最常见的POST/PUT请求中，客户端以Content-Type表头告诉服务器实际发送的数据类型，同样，在响应中，告诉客户端返回的内容的类型。
如果没有指定它，默认为 text/html。

媒体类型(Media-Type)

HTTP的媒体类型是根据「多用途互联网邮件扩展类型」衍生而来。但由于Web与Email有所不同，所以两种类型也有差异。

媒体类型的格式

由类型/次要类型以及可选参数所构成。其中，最常见的可选参数是charset:

其中，类型(type)主要分为五大类：

  1. 文字(text)
  2. 图片(image)
  3. 音频(audio)
  4. 视频(vedio)
  5. 应用(application)

其中「application/x-www-form-urlencoded」发送的数据会被编码为 key:value 数值对，比如 name=shiddong&position=shanghai。
而在现代化的前端框架中，提交表单数据的方式主要以json为主，尽量默认使用application/json.

另外还有两种复合类型：

  1. 多类型(multipart)
  2. 消息(message)

其中「multipart/form-data; boundary=—-XXXXXX」主要是HTML表单上传文件时使用。
boundary 用于分割数据。当文件太长，HTTP无法在一个包之内发送完毕，就需要分割数据，分割成一个一个chunk发送给服务端。

处理方式

无论是请求还是响应，若信息中携带有payload body，都应该生成Content-Type表头。

nosniff

在响应后，浏览器会在某种情况下进行MIME查找，但是并不一定遵循Content-Type的值，为了防止这种行为，可以将设置为 nosniff:

它相当于一个提示标志，被服务器用来提示客户端一定要遵循在Content-Type首部中对MIME类型的设定，而不能对其进行修改。这就禁用了客户端的MIME类型嗅探行为，也就是意味着网站管理员确定自己的设置没有问题。
